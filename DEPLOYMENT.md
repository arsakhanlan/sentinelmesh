# SentinelMesh — public-demo deployment guide

> One VM. One deploy script. Two HTTPS subdomains. Same demo experience as
> local, just on a public URL.

End state after this guide:

- `https://sentinel.<your-domain>` — the SOC dashboard, Policy Lab, Microsoft AI page, all live
- `https://skynest.<your-domain>` — the AI-concierge booking site
- All adversary scenarios fire, the live-agent input runs real LangGraph plans, approvals are clickable, audit chain is intact
- Caddy auto-renews TLS certificates, rate-limits abusive clients, and injects the API key server-side so the JS bundle ships zero secrets

Total time from a fresh VM to the live URL: **about 30 minutes**, of which 5 are typing values into one env file and 25 are waiting for builds.

---

## 0. What you need before the script can run

The script (`ops/scripts/deploy.sh`) installs Docker, configures the firewall, generates secrets, builds, and brings up the stack. It can't do these four things for you:

| Pre-requisite | How |
|---|---|
| **A VM** | 4 GB RAM, 2 vCPU, 40 GB disk, Ubuntu 24.04 LTS. Tested combos in §1. |
| **A domain** | Anything you control DNS for. The deploy script uses two subdomains: `sentinel.<domain>` and `skynest.<domain>`. |
| **Two A records** | Both subdomains pointing at the VM's public IP. The script verifies this and aborts if DNS isn't set up. |
| **A DeepSeek (or OpenAI-compatible) API key** | DeepSeek's `sk-...` key works directly. OpenAI / Groq / Together / OpenRouter / Azure OpenAI all work the same way — they all use the OpenAI-compatible `/v1/chat/completions` shape. |

That's it. Everything else (every secret, every Postgres password, every JWT signing key) is generated by the script.

## 2. Set the DNS records

At your domain registrar (Cloudflare / Porkbun / Namecheap / etc.), add two A records:

```
sentinel    A   <vm-public-ip>
skynest     A   <vm-public-ip>
```

If you're using Cloudflare, **turn the orange proxy cloud off** (set to "DNS only"). Caddy must reach Let's Encrypt directly to obtain certs; Cloudflare's proxy interferes with the HTTP-01 challenge.

DNS propagation takes anywhere from 30 seconds to 30 minutes. Verify before running the script:

```bash
# From your laptop:
dig +short sentinel.<your-domain>
dig +short skynest.<your-domain>
```

Both should return the VM's IP.

## 4. Run the deploy script

```bash
sudo bash ops/scripts/deploy.sh
```

What happens, in order:

1. **Installs prereqs** — Docker engine, Compose plugin v2.27+, git, openssl, jq, dnsutils. Idempotent: skipped if already installed.
2. **Configures the firewall** — opens 22 (SSH), 80 (HTTP), 443 (HTTPS + UDP for HTTP/3).
3. **Generates `/etc/sentinelmesh.env`** — pre-fills every auto-generatable secret (Postgres password, three API keys, JWT secret) with `openssl rand -hex 32`.
4. **Pauses for you to edit four fields** — see §5 below.
5. **Bcrypts the admin password** — converts `SM_ADMIN_PLAINTEXT_PASSWORD` into `SM_ADMIN_BASIC_AUTH` and clears the plaintext.
6. **Validates the env** — refuses to continue if anything is still `REPLACE_ME` or `your-domain.example`.
7. **Verifies DNS** — checks both subdomains resolve to this VM's public IP. Aborts with a clear error if not.
8. **Builds and starts** — `docker compose build --pull && up -d` with the prod overlay.
9. **Waits for Let's Encrypt certs** — up to 3 minutes; usually ~30 seconds.
10. **Smoke tests** — curls both subdomains, the metrics summary API, and the demo-site `/health`.
11. **Prints the URLs and the day-2 commands.**

---

## 5. The four fields you have to fill in

When the script pauses and opens nano on `/etc/sentinelmesh.env`, change these four values:

```diff
- SM_DOMAIN=your-domain.example
+ SM_DOMAIN=acmebooking.dev

- SM_ACME_EMAIL=you@your-domain.example
+ SM_ACME_EMAIL=ops@acmebooking.dev

- OPENAI_API_KEY=sk-deepseek-REPLACE_ME
+ OPENAI_API_KEY=sk-deepseek-aBcDeF1234567890...

- SM_ADMIN_PLAINTEXT_PASSWORD=REPLACE_ME_pick_a_strong_admin_password
+ SM_ADMIN_PLAINTEXT_PASSWORD=4hY!8qPz#xL2vN9w
```

Save (Ctrl-O, Enter) and exit (Ctrl-X). The script picks up where it left off.

That's the **only** manual editing you have to do. Everything else in that file (the five auto-generated secrets, the LLM base URL, the model name, the CORS list, the Caddy basic-auth hash) is filled in by the script.

If you want to use OpenAI instead of DeepSeek: also change two more lines in the same file before saving:

```diff
- OPENAI_BASE_URL=https://api.deepseek.com
+ OPENAI_BASE_URL=https://api.openai.com

- OPENAI_MODEL=deepseek-chat
+ OPENAI_MODEL=gpt-4o-mini
```

## 7. Day-2 operations

All run from the repo root on the VM:

```bash
sudo bash ops/scripts/deploy.sh status      # docker compose ps
sudo bash ops/scripts/deploy.sh logs        # tail every container
sudo bash ops/scripts/deploy.sh logs caddy  # tail one
sudo bash ops/scripts/deploy.sh restart     # restart all containers in place
sudo bash ops/scripts/deploy.sh upgrade     # git pull + rebuild + restart
sudo bash ops/scripts/deploy.sh backup      # run a one-shot backup
sudo bash ops/scripts/deploy.sh stop        # stop containers (preserve volumes)
sudo bash ops/scripts/deploy.sh down        # stop + remove containers (preserve volumes)
sudo bash ops/scripts/deploy.sh test        # re-run the smoke tests
```

### Set up the nightly backup cron (one-time)

```bash
sudo install -m 0755 ops/scripts/backup.sh /usr/local/sbin/sentinelmesh-backup
echo '0 3 * * * /usr/local/sbin/sentinelmesh-backup >> /var/log/sentinelmesh-backup.log 2>&1' \
    | sudo crontab -
```

### Off-host backup sync (optional, recommended)

Install `rclone`, point it at a B2 bucket / S3 bucket / another rsync target, then:

```bash
sudo tee /etc/sentinelmesh-backup.env > /dev/null <<'EOF'
SM_BACKUP_REMOTE=b2:my-bucket/sentinelmesh
EOF
sudo chmod 600 /etc/sentinelmesh-backup.env
```

The backup script picks up `SM_BACKUP_REMOTE` and runs `rclone copy` after the local dump.

### Wipe demo data (manual reset)

```bash
sudo bash ops/scripts/deploy.sh stop
docker volume rm sentinelmesh_pgdata sentinelmesh_skynest_data
sudo bash ops/scripts/deploy.sh restart
```

## 8. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Caddy can't get a certificate | DNS not yet propagated, OR Cloudflare proxy is on | `dig +short sentinel.<domain>` should return the VM IP. Set Cloudflare to "DNS only". Wait, then `deploy.sh restart`. |
| `port 443 already in use` | Another web server is running | `sudo lsof -i :443` — disable nginx / apache. |
| SOC dashboard loads but Threat Feed says "Disconnected" | WebSocket isn't routing | `docker compose logs caddy` and `docker compose logs backend`. The `wss://sentinel.<domain>/ws/...` path must reach backend:8080. |
| `/api/...` returns 401 from the browser | Caddy isn't injecting the API key | `curl -fsSL https://sentinel.<domain>/api/v1/metrics/summary` should return JSON with no `X-API-Key` header from your end. If it returns 401 Caddy isn't applying the `request_header X-API-Key` directive — re-check `Caddyfile` is mounted. |
| Live-agent runs hang on "planning…" | LLM key is wrong / rate-limited | `docker compose logs agents`. DeepSeek returns clear error messages. Verify `OPENAI_API_KEY` and `OPENAI_BASE_URL` in `/etc/sentinelmesh.env`. |
| OOM kills | VM has < 4 GB RAM | Either resize the VM, or set `AGENT_LLM_MODE=stub` in `/etc/sentinelmesh.env` (drops the LLM container's footprint). |
| Bot traffic drives up costs | Rate-limit too generous, or OpenAI cap not set | Tighten Caddy `rate_limit` blocks (in `ops/caddy/Caddyfile`), and set a hard $5/mo soft cap on the OpenAI / DeepSeek dashboard. |

## 10. What this deployment does NOT include

- **Single sign-on** for the SOC dashboard. Public visitors get the same access via Caddy-injected key. If you need real per-user auth (so each operator has their own approval queue), wire OIDC / GitHub OAuth at the Caddy layer with `caddy-security`. Out of scope for the portfolio demo.
- **Multi-region failover.** Single VM. If it goes down, the URL is down. If you want HA, port the same compose to AKS / ECS / GKE — same containers, same Caddyfile (replaced by an ingress controller).
- **Proper audit-log durability.** Postgres is on the VM's local disk; backups are nightly. For a real production deployment, point at a managed Postgres (RDS / Azure Database / Supabase).
- **Real PII handling.** SkyNest stores fake bookings; nothing sensitive. If real data is added, regulatory review is needed (see DLP / consent).

These are deliberate omissions for a portfolio-grade demo. None of them block the live URL working identically to local.
